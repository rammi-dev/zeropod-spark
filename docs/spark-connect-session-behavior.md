# Spark Connect Session Behavior (relevant to ZeroPod checkpoint/restore)

## Key Finding: Sessions Are NOT Tied to TCP Connections

Spark Connect uses a **session ID** (UUID) sent with every gRPC request.
The server stores sessions in a `ConcurrentHashMap<SessionKey(userId, sessionId), SessionHolder>`.
Sessions persist in server memory independently of the gRPC/TCP connection.

### What SessionHolder stores (server-side, in JVM heap)
- SparkSession object (SQL config, catalog)
- Temp views (created via `createOrReplaceTempView`)
- Cached DataFrames (`dataFrameCache` — ConcurrentMap)
- ML model cache
- Registered UDFs
- Streaming query listeners
- Plan cache for logical plans
- Active/inactive operation tracking

### Session lifecycle
- **Created** on first request with a given `(userId, sessionId)`
- **Retrieved** on subsequent requests matching the same key — no TCP affinity
- **Expired** by background maintenance thread after inactivity timeout
- **Released** explicitly via `ReleaseSession` RPC (with optional `allow_reconnect=true`)

### Dual session ID mechanism
- **Client `session_id`**: UUID generated by PySpark (or provided via connection string)
- **Server `server_side_session_id`**: UUID of the underlying SparkSession, returned in responses
- Client sends back `client_observed_server_side_session_id` to detect server-side session changes

## Implications for ZeroPod Checkpoint/Restore

### The flow
1. Notebook creates `SparkSession` → PySpark generates `session_id` UUID
2. User runs queries, creates temp views, caches DataFrames → stored in `SessionHolder` in JVM heap
3. 5 min idle → ZeroPod checkpoints driver → TCP connection is killed (CRIU `AllowOpenTCP: false`)
4. JVM heap frozen to disk by CRIU — `SessionHolder` map preserved
5. User runs next cell → PySpark client gets connection error
6. New TCP SYN → ZeroPod eBPF redirector → activator triggers CRIU restore (~250ms)
7. JVM heap restored — `SessionHolder` map back in memory
8. Client reconnects with **same `session_id`** → server matches to existing session
9. **Temp views, cached DataFrames, UDFs — all still there**

## PySpark Client Retry Behavior — Built-in Automatic Reconnection

The PySpark Spark Connect client has a **built-in retry mechanism** (`DefaultPolicy`) that will
automatically retry on connection failures:

- **Retries `grpc.StatusCode.UNAVAILABLE`** — this is exactly what happens when the TCP connection
  is broken (ZeroPod checkpoint kills TCP, client gets UNAVAILABLE on next call)
- **15 retries** with exponential backoff (50ms initial, 4x multiplier, 60s max, 500ms jitter)
- **Total retry window: ~10 minutes** — more than enough for ZeroPod restore (~250ms)
- **Reattachable execution** enabled by default — long-running queries can resume after reconnect

### Interaction with ZeroPod idle detection

**Critical question: will gRPC keepalives prevent ZeroPod from scaling down?**

ZeroPod's eBPF `socket_tracker` (in `redirector.c`) calls `track_activity()` on **every ingress
TCP packet** to port 15002 (excluding kubelet probes and localhost). This updates a timestamp
that the scaledown timer checks.

PySpark's Spark Connect client does **NOT** configure gRPC keepalives:
- `GRPC_DEFAULT_OPTIONS` only sets max message sizes
- No `grpc.keepalive_time_ms` or `grpc.http2.min_time_between_pings_ms` set
- gRPC's default behavior with no keepalive config: **no client-side PING frames**

However, this needs validation:
- **gRPC default keepalive is disabled** (infinite) when not explicitly set on the client
- But the **OS TCP keepalive** (`SO_KEEPALIVE`) defaults vary — Linux default is 7200s (2h),
  which is longer than ZeroPod's 5m scaledown, so it won't interfere
- Notebook environments (Jupyter, Databricks) may add their own gRPC channel options

**Conclusion**: With default PySpark config, an idle notebook will generate zero TCP traffic
on port 15002 after the last query completes → ZeroPod's 5m timer will expire → checkpoint.

### What happens when the notebook runs the next cell

1. User runs a cell → PySpark sends gRPC request over the existing channel
2. TCP stack discovers the connection is dead → `UNAVAILABLE` error
3. PySpark `DefaultPolicy` catches `UNAVAILABLE` → retries after ~50ms
4. Retry opens new TCP connection → SYN hits port 15002
5. ZeroPod eBPF redirects to activator → CRIU restore (~250ms)
6. By retry 2 or 3 (~200-800ms), JVM is back, gRPC server listening
7. Client sends same `session_id` → server matches to restored `SessionHolder`
8. **Temp views, cached DataFrames, UDFs — all still in JVM heap**
9. User sees ~300-800ms extra latency on the first cell, fully transparent after that

### Remaining validation items
- Does the `server_side_session_id` check cause issues? (CRIU restores same SparkSession UUID, so should match)
- Does the Spark Connect session inactivity timeout need to be disabled/extended? (must outlast ZeroPod scaledown + frozen time)
- Validate with a real notebook (Jupyter) that no hidden keepalives are sent
- Test with a single long-lived PySpark session (not the current phase2 test which creates a new session)

## Security concern: session_id is a shared secret

The `session_id` can be passed in the connection string:
```
sc://spark-connect-server:15002/;session_id=<uuid>
```

This means **any client that knows a session_id can attach to that session** and access its
temp views, cached DataFrames, UDFs, and configuration. There is no additional authentication
beyond the `(user_id, session_id)` tuple.

**Risk**: In a multi-tenant cluster, a developer who discovers another user's session_id
(e.g., via logs, network sniffing, or brute-force) could hijack their Spark session and
access their data. The session_id is a UUID4 (hard to guess), but it's transmitted in
plaintext in gRPC metadata unless TLS is enabled.

**Mitigations to evaluate**:
- Enable TLS on Spark Connect (encrypts session_id in transit)
- Network policies to restrict which pods can reach the Spark Connect service
- Short session timeouts to limit exposure window
- Spark Connect's `user_id` field — sessions require matching user_id + session_id
